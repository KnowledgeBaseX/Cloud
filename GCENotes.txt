- IAM
	Service to manage Who has which access on what

		- Which -> Access (Permission / Role)
		- What -> Resources / Services

	- Role
		- Collection of permissions grouped together
			- Basic
			- Predefined
			- Custom

Compute
	- Managed
		- Google will manage updates, configuration, patching
	- Unmanaged
		- Customer will manage updates, configuration, patching

	- Compute Engine
		- Unmanaged
		- First point of migration
		- Managed Instance Group (MIG)
			- Adding mgmt layer on top of Compute Engine
			- Can be setup with autoscaling on the basis of CPU utilization metric (default)
			- It has instance template for defining what we need
		- Gradual Deployment
			- Min risk while applying patches
			- has maxSurge and maxUnavailable params to control the behavior
			- recommended value 1 and 0 ~ means add 1 VM without stopping any (0) VM until that one VM is fully operational
	- GKE
		- You come up with container image, K8 configuration
		- Partially managed
		- Container with full control
		- Microservice, API
	- Cloud Run
		- You come up with container image, google will provide everything
		- Fully managed
		- Container with no control
		- Statless app, API, microservices
	- App Engine
		- You come up with source code google will provide everything
		- Fully managed
	- Cloud Run Functions
		- Functions
		- Fully managed
		- Useful for event based arch

GKE
	- Cluster: Collection of Nodes
	- Node: Virual Machine. Runs multiple pods
	- Pod: Smallest deployable unit in Kubernetes. Can have one or more containers
	- Preemptible Nodes ~ Spot nodes
		- Batch jobs
		- Stateless, Can tolerate interuption

	- Autoscaling
		- Cluster Autoscaler
			- Scales # of nodes in the cluster
			- Best practice set min and max nodes
		- Horizontal Pod Autoscaler
			- Scales # of podes
			- Don't restart pods. Add/remove instead
			- Can scale based on CPU/mem
		- Vertical Pod Autoscaler
			- Scales Compute and memory of individual pods
			- May restart pods
			- Can't be done on custom metrics
	- Autopilot
		- Fully manged GKE
	- Kubernetes Secrets
		- To store sensitive data
		- Stored in K8s pods

	- Deploying through Dockerfiles
		- Create Dockerfile
		- Build Image
		- Push image to Artifact Registry
		- Create K8s Deployment manifest which references location of image
		- Use kubectl to reference manifest and deploy ~ this is cube control

Cloud Run
	- Fully managed
	- Containers with no control
	- Containers invocable via HTTP request
	- Stateless, API, microservices
	- Autoscaling
		- Dynamically performs horizontal autoscaling (adding instance)
		- Can scale down to zero (0 instances when no traffic). Remember invoked through HTTP Request


App Engine
	- Fully managed
	- You come up with source code and Google provide runnable env (PAAS)


Cloud Run Functions
	- Fully managed
	- Event driven
	- Autoscaling
		- execution time, memory, cpu
